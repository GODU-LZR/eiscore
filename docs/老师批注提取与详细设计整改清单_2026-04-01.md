## 2. 老师意见归纳

老师的批注意见可以归并为五类：

1. 全文口径问题
   设计说明书应写成“设计类”文本，不宜通篇使用“本文”“本论文”，应改为“本设计”“本系统”。

2. 图表组织问题
   图线条太乱、字号偏小；功能结构图应前置；活动图必须跟在对应的用例规约后；所有表都要有单独说明和正文引用，不能堆叠。

3. 章节职责问题
   功能模块划分属于需求分析，不应放到后面重复；第二章应更偏概述性；概要设计章节标题和小节命名需规范，如“系统概要设计”“系统概要设计目标”“架构”。

4. 数据库设计与详细设计边界问题
   表结构说明与第 5.9 节物理表设计应融合，先讲设计思想，再展示表格；第六章不能重复第五章已经讲过的数据表结构。

5. 第六章详细设计问题
   这是老师最强调的部分。详细设计需要真正回答“系统怎么实现”，而不是继续写需求描述或表结构说明。
   对数据库中心架构来说，详细设计应重点写：
   - 页面如何接收输入
   - 前端如何调用接口
   - PostgREST 如何暴露函数或视图
   - PostgreSQL 函数怎样完成业务处理
   - 权限控制和流程状态如何在调用链中生效
   - 时序图如何对应这个过程

## 3. 对当前稿子的直接影响

### 3.1 必须全文统一替换的口径

需要重点搜索并替换这些词：

- `本文`
- `本论文`
- `论文`

优先改成：

- `本设计`
- `本系统`
- `本说明书`

但要注意不能机械全替换，例如参考文献、图题或固定栏目中仍需按实际语义判断。

### 3.2 第三章需要继续保持的结构

第三章当前应按以下顺序组织：

1. 业务现状与流程分析
2. 总体功能结构图
3. 分模块用例图
4. 对应的用例规约表
5. 对应的活动图
6. 非功能需求

也就是说：

- `总体功能结构图` 必须在各模块用例图前
- `活动图` 必须紧跟相应 `用例规约表`
- 每个表前要有一句说明

### 3.3 第五章应怎么收

第五章不要先大段重复表结构细节，然后又在 `5.9` 重复一遍。更合理的顺序是：

1. 先说某一组表的设计思想和业务作用
2. 再给出对应物理表结构
3. 每张表前有单独说明

例如：

- 先说用户与角色设计为什么这样组织
- 再给出 `用户表`、`角色表`、`用户角色关联表`

### 3.4 第六章必须重写的地方

第六章不能再主要写成：

- “某函数实现某功能”
- “某表有这些字段”
- “这里支持入库和出库”

而要改成“调用链式”表达。老师点名的问题是：
你没有把“前端输入 -> 接口调用 -> 后端处理 -> 权限控制/状态更新”这一整条链路讲清楚。

因此，第六章推荐按以下结构重写：

#### 6.1 详细设计概述

- 说明本系统没有传统 `Controller/Service` 类
- 说明详细设计对象换成：
  - 页面组件
  - 接口入口
  - 数据库函数
  - 核心表
  - 流程运行对象

#### 6.2 用户与权限控制详细设计

不要重复讲表字段。应重点写：

1. 登录后用户信息如何进入前端状态
2. 页面访问如何判断当前用户权限
3. 请求发到 PostgREST 时如何携带身份信息
4. PostgreSQL / RLS 如何限制数据可见范围
5. 某个页面如何完成“前端判断 + 数据库限制”的双层控制
6. 用时序图展示从页面到数据库的权限控制链

#### 6.3 库存入库与出库详细设计

不要再把需求分析中的“为什么要入库出库”重复一遍。应重点写：

1. 入库页面采集哪些输入
2. 如何调用 `stock_in` 类函数
3. 批次号如何生成或复用
4. 库存批次表和库存流水表如何在同一事务中更新
5. 出库时如何校验可用库存
6. 异常如何返回到页面
7. 用时序图表示入库和出库过程

#### 6.4 流程审批与状态映射详细设计

应重点写：

1. 页面怎样发起流程
2. 流程定义、实例、任务记录如何落库
3. 节点推进时如何做权限判断
4. 如何把流程状态写回业务表
5. 时序图如何展示流程启动与状态写回

#### 6.5 轻量化语义模块详细设计

不要只写“语义展示了什么”。应重点写：

1. 表级语义和列级语义数据如何组织
2. 本体关系工作台怎样读取这些语义数据
3. 语义怎样与应用配置、流程和权限解释发生关系
4. 这一模块在系统里具体承担什么设计职责

## 4. 当前最优先的整改顺序

建议按下面顺序处理：

1. 全文搜索，把“本文/本论文”先改成“本设计/本系统”
2. 把第三章图表顺序彻底理顺
3. 把第五章的数据表叙述和 5.9 融合
4. 重点重写第六章，让详细设计从“功能说明”变成“实现链路说明”
5. 最后再统一检查图号、表号、题注和正文引用

## 5. 一句话判断

老师最不满意的不是“图不够多”，而是：

`第六章没有真正写出你的系统是怎么被实现出来的。`

所以后续修改的重心，应当放在把数据库中心架构下的实现链路讲清楚，而不是继续堆功能描述。
