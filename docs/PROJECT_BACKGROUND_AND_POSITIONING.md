# EISCore 项目前置背景与立项定位

文档版本：v1.0  
整理日期：2026-02-08  
来源说明：本文件基于项目负责人提供的立项材料整理，作为后续“文档-代码对照评估”的基线。

## 1. 项目定位

EISCore（Enterprise Information System Core）同时服务于两类目标：
- 学术目标：毕业设计课题验证与论文产出。
- 商业目标：面向中小制造企业的可商用企业信息化底座。

核心服务对象场景：以南派食品为代表的农海产品深加工企业，业务具有“多批次、小批量、原料属性非标”特征。

## 2. 研究背景与意义

### 2.1 背景问题

当前中小制造企业数字化常见痛点：
- 大型 ERP（如 SAP）成本高、实施周期长，不适配中小企业体量。
- 轻量 SaaS 定制深度不足，新增字段、改页面布局常受平台限制。
- 传统三层后端架构运维成本高，中小企业缺少专职后端运维团队。
- 物料管理（MMS）存在台账混乱、批次追溯困难。
- 人员管理（HMS）存在多系统账号割裂、权限治理不统一。

### 2.2 研究目的

构建数据库中心（Database-Centric）的 EISCore 底座，目标是：
- 建立 MMS + HMS 双核能力，强化主数据治理与统一鉴权。
- 以 PostgREST 自动映射 API，减少传统后端接口开发。
- 结合 BPMN 2.0 实现可视化流程设计与流程驱动。
- 探索 WebIDE + 大模型的 AI 开发器，实现自然语言到页面与数据结构的快速生成。

### 2.3 研究意义

- 理论层面：验证“低代码前端 + BaaS + 数据库内逻辑”在中小企业场景下的可行性。
- 工程层面：降低部署与维护复杂度，提升业务变更响应速度。
- 应用层面：实现物料全生命周期追踪、权限统一治理与模块化扩展能力。

## 3. 国内外发展与问题

### 3.1 国外趋势

- 组装式企业（Composable Enterprise）强调模块化与可重组。
- 微前端已成为大型 Web 解耦常见方案。
- “厚数据库”回归，数据库逻辑前移，减少中间层。
- 生成式 UI 工具（如 v0 一类）推动前端自动化生成能力。

### 3.2 国内现状

- 中小企业定制系统仍大量依赖传统后端，迭代成本较高。
- 低代码平台虽普及，但常见闭源协议导致迁移与二开受限。

### 3.3 主要缺口

- 架构偏重：运维门槛高。
- 定制僵化：界面和字段变更成本高。
- 前后端协同成本高：小改动需要跨层联动。

## 4. 研究内容、预期目标、关键问题

### 4.1 主要研究内容

1. MMS 物料管理核心模块：支持非标属性、入库/出库/调整事务逻辑。  
2. HMS 安全体系：统一认证与细粒度权限控制。  
3. 自动 API 网关：数据库结构与函数自动映射 REST API。  
4. BPMN 2.0 流程引擎：流程设计、状态流转、执行闭环。  
5. 全栈 AI 开发器：自然语言到 SQL/Vue/配置的自动化生成探索。

### 4.2 预期目标

- 形成可运行 EISCore 原型并完成南派食品场景验证。
- 关键接口性能目标：P95 < 200ms（目标值）。
- 实现 AI 辅助生成 CRUD 页面与数据库逻辑的原型验证。
- 产出：系统源代码、数据库脚本、用户手册、毕业论文。

### 4.3 拟解决关键技术问题

- 无传统 Service 层下的数据库事务封装与异常回滚设计。
- AI 生成前端与数据库 Schema 的一致性保障。
- 微前端与 PostgREST 接口的统一适配与治理。

## 5. 技术方案与路线

### 5.1 技术架构（初期选型）

- 前端：Vue 3 + TypeScript + Element Plus。
- 微前端：qiankun。
- API 层：PostgREST。
- 数据层：PostgreSQL（PL/pgSQL、RLS、JSONB）。
- 流程：BPMN 2.0。
- AI 开发环境：WebIDE + 大模型能力接入。

### 5.2 技术路线

1. 需求分析与 BPMN/数据模型梳理。  
2. 数据库核心开发（Schema、函数、RLS）。  
3. PostgREST 映射与接口验证。  
4. 微前端基座与通用组件建设。  
5. AI 开发器能力打通与提示词迭代。  
6. 全链路集成测试与原型验证。

## 6. 研究方法

- 文献研究法：数据库中心架构、BPMN、生成式 UI。  
- 案例分析法：围绕目标企业痛点设计方案。  
- 原型迭代法：敏捷交付与现场反馈闭环。  
- 实证研究法：对比传统手写与 AI 生成效率/质量。
