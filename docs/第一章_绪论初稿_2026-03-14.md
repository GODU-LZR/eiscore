# 第一章 绪论

题目：面向湛江中小制造企业的企业信息化系统核心设计与实现——以广东南派食品有限公司为例
版本：初稿
整理日期：2026-03-14
说明：本稿根据当前项目实际进展、中期报告和已核验参考文献撰写，引用编号暂对应《论文最终文献池》中的编号，后续可在整篇论文合稿时统一调整。

## 1.1 研究背景与意义

近年来，数字经济快速发展，数字化转型已经从企业的可选项逐渐演变为提升竞争力和经营韧性的关键路径。从已有研究来看，数字化转型不仅关系到企业的信息系统建设水平，也会影响企业的组织协同、资源配置效率和持续成长能力[1-3]。对于制造业而言，数字化建设已经不再只是单纯地引入几套软件，而是需要围绕生产、仓储、采购、人员和流程等核心业务形成贯通的数据链路[1][3]。同时，制造企业的数字化能力也具有多维度特征，不仅包含技术设施，还涉及组织管理、数据利用和业务适配等方面[4]。

但是，和大型制造企业相比，地方中小制造企业在数字化建设过程中面临的约束更为突出。一方面，中小企业普遍缺乏稳定的信息化投入能力和专业技术团队，难以承受大型 ERP 系统较高的采购、实施和维护成本；另一方面，许多轻量化工具虽然上线较快，却难以随着业务变化做深度定制，最终容易出现“能用但不好用”“上线后难扩展”的问题[5][6]。已有研究指出，中小制造企业的数字化转型并不是简单的软件替换，而是需要结合企业规模、业务流程和资源条件，选择合适的实施路径和平台化支撑方式[5-7]。这说明，面向中小制造企业的信息化系统建设，不能完全照搬大型企业的重型架构，也不能只追求短期可上线，而应在轻量部署、业务适配和后续扩展之间取得平衡。

湛江地区具有较明显的农产品、水产品和食品加工产业特点，当地许多中小制造企业在原料组织方式、生产批次管理和现场业务流程上都具有一定的非标准化特征。以广东南派食品有限公司为例，其业务场景既涉及物料多批次流转、原料属性差异化管理，也涉及人员岗位、权限配置和流程审批等管理需求。如果依旧采用传统的“前端页面 + 独立后端服务 + 分散数据库”的开发方式，一旦企业新增业务字段、修改表单结构或调整流程状态，就往往需要同步修改前端、接口和数据库，开发维护成本较高，也不利于中小企业后续自行扩展。

基于上述背景，本文尝试设计并实现一个面向湛江中小制造企业的企业信息化系统核心平台 EISCore。系统以数据库中心架构为基础，以 PostgreSQL 为核心数据与逻辑承载平台，并通过 PostgREST 自动映射数据接口，减少传统后端接口层的重复开发负担。在前端侧，系统采用微前端方式承载不同业务模块，使人事管理、物料管理、应用中心和移动端功能能够在统一底座上协同运行。与此同时，系统进一步引入 BPMN 流程建模和轻量本体语义增强机制，尝试解决流程协同、数据语义表达和动态扩展之间的一致性问题。

本课题具有一定的理论意义和实践意义。在理论层面，本文将数据库中心架构、微前端、BPMN 和轻量本体语义建模结合起来，探索一种更适合中小制造企业资源约束场景的企业信息化实现路径，也为“低代码前端 + 数据库逻辑集中承载”的应用模式提供了一个具体案例[8][11-13][14-20]。在实践层面，本课题以南派食品的真实业务场景为基础，围绕人事管理和物料管理两个核心域展开建设，能够为企业实现台账追踪、人员治理、流程联动和后续业务扩展提供较低成本的技术支撑。因此，本课题不仅关注系统功能能否实现，也关注系统是否适合在中小制造企业的现实环境中真正落地。

## 1.2 国内外研究现状

### 1.2.1 国内研究现状

从国内研究来看，近年来围绕企业数字化转型、制造业信息化和工业互联网平台赋能的研究明显增多。相关研究普遍认为，数字化转型已经成为企业提高运营效率、增强资源配置能力和实现持续成长的重要手段[1-3]。在制造业领域，研究者更多关注制造企业数字化能力的结构构成、影响因素以及转型实施路径。例如，有研究从能力维度出发，对制造企业数字化能力的结构进行了识别和测量[4]；也有研究专门讨论了中小制造企业在数字化转型中的路径选择与现实制约[5]；针对传统制造工厂的研究则进一步指出，数字化改造不仅是设备联网和数据采集问题，还涉及业务流程再设计和组织协同机制重构[6]。此外，工业互联网平台如何为制造企业提供数据支撑和平台能力，也成为近年来的重要研究方向[7]。

在软件工程和系统实现层面，国内研究对工作流、本体语义和数据库语义建模也进行了较深入的探讨。围绕 BPMN 2.0，已有研究对流程编排的形式语义和分析方法进行了讨论，为业务流程的规范化描述和执行分析提供了理论基础[8]。在工作流约束和流程适配方面，也有学者从可满足性分析、语义修正等角度研究了复杂业务流程在动态条件下的执行问题[9][10]。在本体语义方面，国内研究已经从本体视图映射、关系模型到本体模型的转换、本体推理支持的数据查询等多个层面展开，为业务数据的语义表达和语义增强提供了方法基础[11-13]。

总体来看，国内研究已经分别在制造业数字化、流程建模和本体语义等方面形成了较好的理论积累，但这些成果多数还是分散在不同研究方向中。对于中小制造企业而言，如何把“业务模块建设”“流程协同”“数据语义表达”和“轻量化系统部署”整合到同一套企业信息化平台中，相关研究和工程化案例仍然相对不足，这也正是本文希望进一步探索的地方。

### 1.2.2 国外研究现状

国外研究在软件架构演进和开发模式创新方面起步较早。近年来，低代码和无代码开发逐渐成为企业软件交付的重要方向。相关研究指出，低代码平台能够通过模型驱动、配置化开发和可视化建模等方式缩短系统交付周期，提高业务响应速度[15-17]。在制造业场景中，已有研究尝试将低代码平台用于业务流程自动化，以降低系统开发门槛并增强流程适应能力[14]。这些研究说明，在资源有限且需求变化频繁的场景下，以模型和配置为核心的软件构建方式具有明显优势。

与此同时，前端大型应用的复杂度不断上升，微前端逐渐成为提升系统解耦能力和团队独立交付能力的重要方案。Taibi 等对微前端的基本原则、实现方式和潜在问题进行了总结，指出该架构能够将复杂前端拆分为相对独立的业务单元，但同时也会带来集成复杂度、运行协调和技术治理方面的新挑战[18]。Nishizu 等则从实现机制上讨论了基于组件封装和信号机制的微前端实现方式，为模块隔离和页面集成提供了可参考的技术路径[19]。这类研究对于本文构建微前端基座、拆分人事和物料等业务子系统具有直接借鉴意义。

在后端架构方面，数据库中心化思想重新受到重视。Pirani 等提出，应用接口不一定都要建立在厚重的应用服务器层之上，而可以更多围绕数据模型和数据库能力进行组织，从而形成以数据为中心的应用接口模式[20]。这一思路与本文采用 PostgreSQL 承载核心逻辑、借助 PostgREST 自动映射接口的技术路线高度契合。相较于传统三层架构，数据库中心架构在降低开发重复劳动、减少系统链路长度和增强数据一致性方面具有一定优势，特别适合技术团队较小、运维资源有限的中小企业场景。

总体来看，国外研究已经在低代码、微前端和数据库中心架构方面积累了较成熟的理论与实践经验，但这些研究往往侧重某一项技术本身，对“中小制造企业核心业务系统”这一特定场景的整体整合仍不充分。尤其是在业务流程、权限治理、动态扩展和语义建模同时存在的情况下，如何形成一套兼顾轻量性与可扩展性的企业信息化底座，仍然值得进一步研究。

### 1.2.3 现有研究存在的问题

综合国内外研究现状可以看出，现有研究和系统方案仍存在以下几个方面的不足。

第一，研究成果较为分散。数字化转型研究更多聚焦企业管理和能力提升，工作流研究更多关注流程本身，本体研究则偏重语义建模方法，而真正把这些内容统一到具体企业系统中的研究相对较少。第二，很多成熟方案更适用于大型企业或标准化程度较高的业务场景，对于地方中小制造企业常见的“小批量、多批次、频繁变更”特点支持不足。第三，传统企业系统通常仍依赖较重的后端服务层和接口开发模式，一旦业务字段、表单结构或流程规则发生调整，往往需要多层同步修改，维护成本较高。第四，现有研究对数据语义和流程状态之间的一致性关注不足，在动态应用生成、权限控制和流程驱动协同时容易出现表达不统一的问题。

因此，围绕中小制造企业构建一种能够兼顾轻量部署、流程协同、数据语义表达和业务持续扩展的企业信息化底座，既有现实需求，也有进一步研究的空间。

## 1.3 本文主要研究内容

结合前述问题分析，本文围绕 EISCore 系统的设计与实现展开，主要研究内容包括以下几个方面。

第一，围绕中小制造企业的典型业务需求，构建以人事管理和物料管理为核心的业务底座，重点解决员工信息、组织结构、权限治理、物料主数据、库存台账和出入库管理等问题。第二，采用数据库中心架构，将 PostgreSQL 作为核心数据与业务逻辑承载平台，并借助 PostgREST 自动映射接口，减少传统后端接口层的开发负担。第三，在前端架构上采用微前端方式，对人事、物料、应用中心和移动端等子系统进行拆分与集成，以提升系统的模块独立性和后续扩展能力。第四，引入 BPMN 2.0 流程建模机制，使系统不仅能管理静态业务数据，也能支撑审批流转和状态变化等过程型业务。第五，在系统建设过程中引入轻量本体语义增强机制，通过表级语义、列级语义和表间关系描述，增强系统对业务对象和数据含义的表达能力，为流程、权限和动态应用的协同提供统一基础。

截至目前，系统整体完成度已达到 80% 左右，开题阶段提出的主要功能大部分已经实现。人事、物料、应用中心和移动端等核心模块已形成较完整原型，流程设计和语义增强也已完成阶段性落地，因此本文的研究内容不仅具有方案设计层面的讨论，也具有一定的实现和验证基础。

## 1.4 本文组织结构

本文共分为八章，各章内容安排如下。

第一章为绪论，主要介绍课题研究背景、研究意义、国内外研究现状以及本文的主要研究内容与结构安排。第二章介绍系统开发涉及的关键技术和工具，包括 Vue 3、微前端、PostgreSQL、PostgREST、BPMN 以及本体语义相关技术。第三章进行系统需求分析，围绕用户角色、核心业务流程和非功能需求展开说明。第四章给出系统总体设计，重点说明系统架构、模块划分和关键设计思路。第五章介绍数据库设计与语义增强设计，包括数据模型、主要表结构和本体语义组织方式。第六章对核心模块进行详细设计，分别说明人事管理、物料管理、流程联动和动态应用配置等模块的实现机制。第七章展示系统实现效果，并结合主要页面和业务链路说明系统运行情况。第八章对系统进行测试与分析，并在结论部分总结全文工作、指出存在的不足以及未来改进方向。

## 1.5 本文的特点与创新点

与传统单纯围绕功能模块开发的毕业设计相比，本文的工作主要有以下几个特点。

第一，本文并未采用常见的“前端 + 独立后端 + ORM 服务层”模式，而是尝试以数据库中心架构承载核心数据与业务逻辑，再通过自动接口映射方式向前端提供服务，这在中小制造企业资源有限的场景下具有较强的现实针对性。第二，本文在实现物料管理、人事管理和流程协同的基础上，引入了轻量本体语义增强机制，使系统不只是完成业务数据录入和展示，还能够从语义层面描述业务对象、字段含义和关系结构。第三，本文把流程建模、权限治理、动态应用配置和语义表达放在同一套平台中统筹考虑，力图解决企业信息系统中常见的“功能做出来了，但后续不好扩展”的问题。第四，本文的研究不是停留在概念方案上，而是建立在当前项目已完成约 80% 的系统实现基础上，具有一定的工程验证价值。
