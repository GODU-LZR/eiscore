# 老师批注整理与修改建议

适用稿件：`1 软件1223林志荣初稿v2.5.docx`

## 一、总体判断

老师的批注主要集中在四类问题上：

1. 章节定位不够稳定。
2. 图表放置与章节功能不匹配。
3. 需求分析和数据库设计的论文规范性不足。
4. 设计总说明和详细设计的展开深度不够。

这些问题不是单纯的措辞问题，而是“论文结构怎么摆”的问题。因此，后续修改时不能只改几句话，而要按章节功能重新收口。

## 二、逐条批注分析

### 1. 设计总说明内容太少

老师批注：`内容太少，要增加。`

问题本质：

1. 当前设计总说明偏像“扩展版摘要”。
2. 三段论还不够完整，尤其是第二段系统介绍和第三段章节说明展开不足。
3. 还没有把系统特色、真实业务约束和当前实现边界说透。

修改建议：

1. 第一段写清楚研究背景，不能只写“数字化转型重要”，要落到南派食品这类企业为什么会在台账、权限、流程和批次追溯上出问题。
2. 第二段要明确介绍 EISCore 的核心设计：数据库中心架构、微前端、PostgREST、流程运行、轻量化语义、闪念应用、移动端冷库模式。
3. 第三段用较完整的方式交代各章内容，不要只是一句带过。
4. 总体篇幅建议明显扩充，接近学校要求的一页量。

### 2. 图1-1 缺少明确说明和引用

老师批注：`这个图放在这里想说明什么，在论文中应有相应叙述及引用。`

问题本质：

1. 图虽然画出来了，但正文没有明确解释它的作用。
2. 图和上下文关系不够紧，像“插进去的图”，不是“被论证需要的图”。

修改建议：

1. 在图前加一句总引导，说明图1-1用于概括论文从企业问题到系统实现再到论文结构形成的总体研究路线。
2. 在图后补一小段解释，按“企业约束 -> 需求分析 -> 总体设计 -> 关键技术 -> 系统实现 -> 测试验证 -> 论文形成”的顺序解释图中每一层。
3. 如果正文仍然解释不顺，可以把图1-1往后放，放在 `1.3 本文主要研究内容` 末尾更自然。

### 3. 第二章应以通用技术介绍为主

老师批注：`这一章是一些通用技术的介绍，你对这些技术的改造和融合，建议放在4.5关键设计思路中。这一章基本是概述性的内容。`

问题本质：

1. 第二章现在混入了太多“本系统如何融合这些技术”的内容。
2. 技术概述章和系统设计章边界不清。

修改建议：

1. 第二章保留“技术是什么、在业界有什么用途、为什么适合本课题”。
2. 第二章不要过多展开“闪念应用如何与语义联动”“Agent Runtime 如何编排工具”“冷库模式如何协同接口”等系统级组合逻辑。
3. 这些内容改放第四章 `4.5 关键设计思路`，或者第七章系统实现部分。

### 4. 图2-1 不适合放在第二章

老师批注：`这里给出技术的详细设计不适合，应放在在系统设计中。`

问题本质：

1. 图2-1 不是单纯的“关键技术介绍图”，而更像“系统技术组织关系图”。
2. 这张图已经带有明显架构设计意味，属于第四章内容。

修改建议：

1. 将图2-1 从第二章移到第四章。
2. 更合适的位置是第四章 `4.3 系统总体架构设计` 后，或者 `4.5 关键设计思路` 前。
3. 第二章中只保留技术文字介绍，不保留这张图。

### 5. 图2-2 放置不合适

老师批注：`这个图放着也不合适。`

问题本质：

1. 图2-2 讲的是“闪念应用、智能体、轻量语义、数据表格应用”的联动。
2. 这种图不属于通用技术概述，也不适合作为第二章的普通技术图。

修改建议：

1. 不建议继续放在第二章。
2. 有两个更合适的位置：
   - 第四章 `4.5 关键设计思路`
   - 第七章“应用中心模块实现”之后
3. 如果第四章强调的是架构和模块，可以把它放第七章，这样更像“实现联动效果图”。

### 6. 第三章功能需求分析缺少总体功能结构图和分模块用例

老师批注：`需求分析应有总体功能结构图，并分模块给出用例图和用例规约。`

问题本质：

1. 现在第三章虽然写了很多文字，但图不足。
2. 用例图已经有部分素材，但还没有完全按“总体 + 分模块”方式组织。
3. 用例规约和活动图还不够。

修改建议：

1. 在 `3.5 功能需求分析` 前面增加一张“系统总体功能结构图”。
2. 再按模块拆开：
   - 基础平台与主数据
   - 物料与仓储
   - 应用中心与流程
   - 移动端
3. 每个模块至少给一个用例图。
4. 选两个最关键流程补用例规约：
   - 采购入库 / 来料检验
   - 生产领料 / 补料

### 7. 第五章物理表结构标题与正文混在一起

老师批注：`没有这么长的标题，正文内容和标题不要混了。`

问题本质：

1. 你的小节标题本身是正常的，如 `5.9.1 用户表 public.users`。
2. 真正的问题是标题下面的说明性正文写法看起来像和标题连在一起，版式不干净。
3. 另外物理表结构展示方式还不够“说明书化”。

修改建议：

1. 保持标题简洁，只保留：
   - `5.9.1 用户表`
   - `5.9.2 角色表`
2. `public.users`、`public.roles` 这类表名放进正文第一句，不放在小节标题里。
3. 每个表建议统一写法：
   - 一句“表X-X展示了……”
   - 一句“该表用于……”
   - 再接字段表
