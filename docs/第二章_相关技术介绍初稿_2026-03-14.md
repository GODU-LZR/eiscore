# 第二章 相关技术介绍

题目：面向湛江中小制造企业的企业信息化系统核心设计与实现——以广东南派食品有限公司为例
版本：初稿
整理日期：2026-03-14
说明：本章根据当前项目仓库中的实际技术栈整理，优先描述系统真正使用到的技术与工具，而不是只重复开题阶段的计划方案。正文中的引用编号暂对应《论文最终文献池》中的编号，后续可在整篇论文定稿时统一调整。

## 2.1 前端开发框架

本系统前端总体采用 `Vue 3 + Vite` 的开发模式。Vue 3 采用组件化思想构建页面，将复杂业务拆分为相对独立的视图模块、表单模块和交互模块，便于后续维护和复用。对于企业信息化系统而言，页面往往包含大量表单录入、列表展示、对话框交互和状态切换逻辑，如果继续使用传统的页面脚本堆叠方式，随着业务模块增加，代码可维护性会迅速下降。因此，组件化前端框架能够在较大程度上提升页面开发效率和结构清晰度。

在工程构建方面，项目采用 Vite 作为前端开发与打包工具。Vite 在开发阶段通过原生 ES Module 机制实现快速启动和热更新，适合本系统这种由多个子应用组成的前端工程。结合仓库中的实际情况可以看到，基座应用、人事子应用、物料子应用、应用中心和移动端等模块都采用了 Vite 作为统一的前端构建工具，这有利于保持各子系统在开发体验和打包方式上的一致性。

除核心框架外，系统还结合业务需求引入了多个常用前端支撑库。其中，`Vue Router` 用于处理前端路由跳转，`Pinia` 用于状态管理，`Axios` 用于和 PostgREST 及其他服务接口进行通信。这几类技术共同构成了前端基础运行框架，使页面状态、用户会话、接口调用和导航跳转能够较稳定地协同工作。

## 2.2 微前端技术

随着系统规模扩大，单体前端应用容易出现模块边界模糊、构建体积膨胀和多人协作冲突等问题。为解决这些问题，本文在前端总体结构中引入了微前端思想。相关研究指出，微前端的核心目标是将复杂前端拆分为多个可相对独立开发、部署和演进的子应用，从而提升大型前端系统的可维护性和团队协作效率[18][19]。

本文在前端总体结构中选用了 `qiankun` 作为微前端框架。选择该技术的原因在于，企业信息化系统的页面通常不会长期固定不变，而是会随着业务扩展不断新增模块、页面和入口。微前端能够为这类系统提供更清晰的边界划分方式，使不同功能域在保持统一入口的同时，相对独立地演进。至于本系统如何围绕微前端组织具体业务子应用、如何与应用中心协同，这部分属于系统总体设计的内容，后文将在第四章进一步展开。

## 2.3 界面组件与交互支撑技术

企业信息化系统在日常使用中，最常见的页面形式是表格、表单、审批视图、统计图表和移动端录入页面。为了提高开发效率并保持界面风格统一，本系统在桌面端主要选用 `Element Plus` 作为基础组件库，在移动端选用 `Vant` 作为主要组件支撑。

`Element Plus` 提供了表单、表格、弹窗、树形控件、分页、消息提示等常用组件，比较适合后台管理类系统的开发需求。由于本项目的人事管理、物料管理和应用中心都包含大量管理界面，因此使用成熟组件库能够减少重复封装工作。对于需要展示大批量数据的页面，本系统还引入了 `AG Grid`，用于支持复杂表格、筛选排序、列配置等交互需求，这对库存台账、员工信息列表和数据应用配置场景都较为重要。

此外，系统还使用了 `ECharts` 进行统计图展示，使用 `html2canvas` 支持部分页面导出和截图场景，使用 `Monaco Editor` 作为应用中心中的代码编辑器支撑，使用 `Markdown-it`、`Mammoth` 等库处理文本与文档内容。这些技术并不是论文研究的主体，但它们在实际系统实现中承担了必要的工具角色，使系统功能能够完整落地。

## 2.4 PostgreSQL 数据库技术

数据库是本系统最核心的技术基础。与传统企业系统将主要业务逻辑集中在应用服务器不同，本文采用以数据库为中心的实现思路，将 PostgreSQL 不仅作为数据存储平台，也作为部分业务逻辑、权限控制和流程状态承载平台。相关研究表明，以数据模型和数据库能力为中心组织应用接口，可以在一定程度上减少中间层重复逻辑，提高系统链路效率，并增强数据一致性[20]。

之所以选择 PostgreSQL，主要有以下几个原因。第一，PostgreSQL 具备成熟的事务支持、视图、触发器、函数和 JSONB 等能力，适合同时处理结构化数据和部分动态属性数据。第二，本系统中的物料管理场景具有“原料批次多、属性差异大”的特点，一些非标准化字段如果完全依赖固定表结构处理，会使设计非常僵硬，而 PostgreSQL 的 JSONB 能够为这类扩展属性提供较高的灵活性。第三，系统中的权限、流程、语义信息以及动态应用配置，本质上都需要和核心业务数据保持一致，将部分逻辑下沉到数据库有助于统一治理。

结合当前项目实现可以看到，系统数据库中已经包含 `hr`、`scm`、`workflow`、`app_center`、`app_data` 等多个 Schema，用于分别承载人事、供应链、流程、应用中心和动态数据应用等内容。同时，项目中还编写了多组 SQL 脚本用于初始化表结构、修复业务逻辑、补充语义信息和扩展流程能力。这说明 PostgreSQL 在本课题中并不只是一个被动存储层，而是整个系统的核心基础设施。

## 2.5 PostgREST 与接口自动映射技术

传统前后端分离系统通常需要单独开发后端接口层，由服务端负责将数据库操作封装为 API。对于中小企业定制系统而言，这种模式虽然常见，但会带来较高的开发与维护成本。本文引入 `PostgREST`，尝试通过数据库模式和权限规则自动映射 RESTful 接口，减少手工编写大量增删改查接口的工作量。

PostgREST 的基本思路是直接把数据库中的表、视图和函数按照约定暴露为 HTTP 接口，使前端能够在具备权限控制的前提下直接访问数据库资源。这种方式与数据库中心架构相匹配，能够明显缩短“页面请求 - 服务处理 - 数据返回”的链路长度，也降低了后端样板代码的数量。对于本系统而言，数据应用配置、表格查询、动态表单保存以及部分流程相关的数据访问，都可以借助这种自动映射方式实现。

但需要说明的是，使用 PostgREST 并不意味着系统完全没有业务约束。相反，只有在数据库层已经把表结构、视图、函数和权限设计清楚的前提下，自动映射接口才是可控的。因此，PostgREST 在本项目中的价值，不是简单替代后端，而是倒逼系统把数据模型和权限边界设计得更清晰、更规范。

## 2.6 行级安全控制与权限治理技术

对于企业信息系统来说，权限控制是核心问题之一。特别是在人事管理、审批流转和业务台账等场景中，不同角色能够看到什么数据、能否修改某些记录、能否执行某个操作，都直接关系到系统安全性。PostgreSQL 提供的 `Row Level Security` 能够在数据库层面对不同行的访问进行限制，从而把数据权限控制从应用代码扩展到数据访问底层。

本系统在人事模块和应用中心中已经大量使用了基于角色和请求声明的权限控制逻辑。例如，项目中的 SQL 脚本针对应用中心、工作流审批和权限语义等内容配置了多条 RLS 策略，并结合 JWT 声明字段判断当前用户角色。这种方式的优势在于：即使多个前端页面或子应用都通过同一个数据接口访问数据库，只要底层策略一致，权限边界就不会因为页面实现不同而发生偏差。

此外，项目还在传统角色权限的基础上，进一步尝试用轻量本体语义方式表达权限含义、流程状态和操作语义之间的关系。相关研究表明，本体映射和语义推理能够为复杂数据对象提供更清晰的语义表达基础[11-13]。虽然当前系统还没有实现完整意义上的复杂知识推理，但通过表级语义、列级语义和权限语义视图，已经初步建立了“数据结构 - 权限表达 - 状态流转”之间的统一描述方式。这也是本课题在系统治理层面较有特点的一部分。

## 2.7 BPMN 工作流建模技术

制造企业的信息化系统并不只是记录静态数据，很多业务场景都包含提交、审核、回退、完成等状态变化过程。例如入库审批、出库审核、请假审批、任务分派等，都属于过程型业务。为了让这些过程能够被可视化建模并与业务数据联动，本文在系统中引入了 BPMN 2.0 工作流建模思想。

已有研究指出，BPMN 2.0 在业务流程描述和流程语义分析方面具有较好的规范性和可表达性[8]；而在复杂工作流执行过程中，流程约束、任务分配和流程修正也是影响系统可用性的重要问题[9][10]。基于这一认识，系统在应用中心中设置了工作流设计器，并在数据库中建立了 `workflow.definitions`、`workflow.instances` 等表，用于存储流程定义和流程运行实例。同时，系统还通过状态映射表把 BPMN 节点与业务数据状态字段关联起来，使流程节点变化能够驱动具体业务记录的状态变化。

从技术定位上看，BPMN 2.0 在本课题中首先是一种规范化的流程表达方式。它使原本分散在页面逻辑、条件判断和人工沟通中的审批过程能够被单独建模，并为后续流程定义、任务分派和流程状态管理提供统一描述基础。至于它与应用中心、审批中心和业务状态写回的具体协同方式，属于系统设计层面的内容，后文将在总体设计和详细设计章节继续说明。

## 2.8 轻量本体语义增强技术

在项目推进过程中，单纯依赖表名、字段名和页面配置，已经难以满足系统持续扩展的需要。尤其是在动态应用配置、流程状态映射和权限联动逐渐增多之后，如果没有统一的语义描述机制，不同模块之间很容易出现“字段能用，但含义不统一”的问题。基于此，本文在系统中引入了轻量本体语义增强机制。

本体相关研究普遍关注概念、属性、关系以及语义映射问题[11-13]。其中，关系模型到本体模型的映射研究，对于本系统特别有借鉴价值，因为本系统本身就是围绕关系数据库构建起来的[12]。在当前项目中，轻量本体语义主要体现在三个层面：一是对业务表进行表级语义标注，明确某张表所代表的业务对象；二是对字段进行列级语义描述，增强字段含义的可解释性；三是通过本体关系工作台展示表与表之间的语义关系，并尝试与流程状态和权限表达进行兼容联动。

需要说明的是，本课题中的“本体论”并不是要做一套完全独立的知识图谱系统，而是作为企业信息系统中的一种轻量语义增强机制存在。它的目标不是替代关系数据库，而是在现有关系数据库和业务功能之上补充更清晰的业务含义表达，使表、字段和关系在系统演进过程中保持较好的可解释性。至于轻量语义如何与应用配置、流程状态和受控工具链联动，这部分属于本系统的融合设计问题，应在总体设计章节展开，而不是在本章概述中过度提前。

## 2.9 容器化部署与运行环境

为了提高系统的可部署性和环境一致性，本文采用 `Docker Compose` 对核心服务进行容器化编排。通过这种方式，可以把数据库初始化、接口服务启动和运行时依赖统一组织起来，降低手工搭建环境时出现的差异。对于中小企业场景而言，部署方式是否清晰、后续是否容易复现，往往和功能实现本身同样重要。

在前端开发运行方面，项目同时使用 `PM2` 维护多个前端子应用进程，以减少手工逐一启动和维护带来的管理成本。该工具本身属于运行环境和工程管理工具，而不是系统设计主体，因此本章只说明其技术用途，不展开具体子应用协同方式。

从开发环境角度看，项目当前主要使用 `Node.js`、`npm`、`Vite` 和 `Docker`。数据库版本为 `PostgreSQL 16`，前端主框架为 `Vue 3`。此外，项目还引入了基于浏览器的 `code-server` 作为轻量 WebIDE 运行环境，用于支持部分开发与调试场景。这部分内容虽然偏工程实现，但对于说明系统整体运行环境和工具链仍然是必要的。

## 2.10 本章小结

本章围绕系统实现所依赖的关键技术进行了概述，重点回答“系统使用了哪些关键技术，它们分别解决什么类型的问题”。整体来看，Vue 3、Vite 和 qiankun 为前端组织方式提供支撑，PostgreSQL、PostgREST 和 RLS 为数据访问与权限治理提供基础，BPMN 2.0 为流程型业务表达提供工具，轻量语义增强机制为表与字段的业务解释提供补充，而 Docker Compose、PM2 等工具则保证系统运行环境的统一和可维护。上述技术共同构成了 EISCore 的技术基础，至于这些技术在本系统中的改造、融合和组织方式，则放到第四章总体设计中进一步说明。
