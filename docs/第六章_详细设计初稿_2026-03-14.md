# 第六章 详细设计

题目：面向湛江中小制造企业的企业信息化系统核心设计与实现——以广东南派食品有限公司为例
版本：初稿
整理日期：2026-03-14
说明：本章在总体设计和数据库设计的基础上，进一步对系统中已经完成实现的关键功能进行详细设计说明。为保证内容与当前项目实际进度一致，本文重点展开用户与权限、库存出入库、流程审批联动以及轻量化语义模块四个部分。

## 6.1 详细设计概述

详细设计阶段的任务，是把总体架构和模块划分进一步落实为可执行的页面、数据流和数据库处理逻辑。与传统软件工程中“详细设计完全围绕类方法展开”的写法不同，EISCore 采用数据库中心架构，因此详细设计不再以 `Controller-Service-DAO` 这一套传统后端类分层为主要表达对象，而是同时围绕前端页面、PostgREST 接口、数据库函数、核心业务表和流程运行对象展开。

结合当前论文项目的实现情况，本章重点说明以下四类内容：第一，用户与权限控制的详细设计；第二，物料管理中库存入库与出库功能的详细设计；第三，应用中心中流程审批与状态映射的详细设计；第四，轻量化语义模块的详细设计。之所以选择这四部分，是因为它们既对应开题阶段提出的核心目标，也是在当前项目中证据最充分、最能体现系统特色的内容。

### 6.1.1 数据库中心架构下的详细设计对象

在本课题中，详细设计的基本单元并不是“类及其成员方法”，而是“页面组件 + 接口入口 + 数据库函数 + 核心表 + 流程运行对象”的组合。换句话说，传统系统中由后端类承担的许多职责，在 EISCore 中被下沉到了 PostgreSQL 函数和数据表约束中。因此，若仍然完全按类图方式展开，反而不能准确反映本系统的真实实现结构。

表6-1 数据库中心架构下的详细设计对象

| 设计对象 | 主要载体 | 在本文中的表现 |
| --- | --- | --- |
| 页面交互对象 | Vue 页面、表格、抽屉表单、审批中心界面 | 负责参数采集、结果展示和流程上下文承载 |
| 接口入口对象 | PostgREST 自动映射接口、`/rpc/*` 调用入口 | 负责把前端请求转入底层函数 |
| 业务处理对象 | `scm.stock_in`、`scm.stock_out`、`workflow.start_workflow_instance` 等函数 | 负责事务、校验、状态更新和结果返回 |
| 数据持久化对象 | `inventory_batches`、`inventory_transactions`、`workflow.instances`、`workflow_state_mappings` 等表 | 负责保存业务状态、流程状态和映射关系 |
| 权限与运行对象 | JWT 声明、RLS 策略、任务分派规则、草稿上下文 | 负责控制谁可以访问、谁可以审批以及业务如何继续流转 |

### 6.1.2 本章的表达方式

基于上述架构特点，本文的详细设计采用以下三种方式展开：第一，用关键函数的输入、输出与处理步骤说明底层业务逻辑；第二，用关键数据表和状态映射说明数据如何落地；第三，用时序图展示页面、接口、数据库函数和业务表之间的运行先后关系。这样处理后，即使系统没有传统意义上的后端类，仍然能够清楚表达各处理环节之间的协作方式。

## 6.2 用户与权限控制详细设计

### 6.2.1 设计目标

用户与权限控制模块主要服务于系统登录身份识别、角色分配、菜单访问约束和数据库数据访问边界控制。对于企业信息化系统而言，仅依赖前端隐藏菜单并不足以保证安全，因此本系统采用“前端鉴权 + 数据库权限控制”相结合的方式，使用户在界面访问和数据访问两个层面都受到约束。

### 6.2.2 结构组成

该部分设计主要由以下几个层面组成：

1. 用户基础数据层：由 `public.users`、`public.roles`、`public.user_roles`、`public.departments`、`public.positions` 等表组成。
2. 前端身份感知层：由人事子应用和其他子应用中的 `user store`、权限工具函数、路由控制逻辑组成。
3. 数据访问控制层：由 PostgreSQL 的 `RLS` 策略与 JWT 声明共同构成。

其核心思想是：用户先在应用层完成身份识别，再由数据库层进一步判断该用户是否具备对特定数据进行查询、写入或更新的资格。

### 6.2.3 前端权限处理流程

在前端层面，系统以用户信息状态作为多个子应用的共同输入。用户登录后，系统会把用户名、角色和相关权限写入用户状态管理模块；随后，页面根据路由元信息和权限判断函数决定当前用户是否可以访问对应功能页面；当用户进入页面后，按钮级操作如新增、修改、导出、配置等也会通过权限函数进行控制。

以人事子应用为例，其页面组织已经包含员工列表、组织结构、用户管理和考勤页面，并通过权限判断函数对页面功能进行细分控制。这种做法的优点在于：一方面，界面行为清晰；另一方面，即使前端误放开了某个入口，数据库层仍然会通过权限策略拦截不合法请求。

### 6.2.4 数据库权限控制设计

数据库权限控制是本部分设计的重点。系统在 `app_center`、`workflow`、`scm` 等关键 Schema 中均启用了 RLS，并结合 `request.jwt.claims` 中的角色信息判断用户的真实访问权限。例如，在流程定义、流程实例和任务分派表上，仅具备高权限角色的用户才允许执行插入、更新和删除操作；在应用中心中，发布路由和状态映射等关键配置表也通过策略限制写入行为。

这种设计的意义在于：权限控制不再完全依赖前端界面，而是直接下沉到数据库层。即使不同子应用共同访问同一张表，系统仍然能够保证数据边界的一致性。

### 6.2.5 关键数据表设计说明

用户与权限控制部分的关键数据表包括：

1. `public.users`：保存系统用户账号、姓名、部门、岗位和状态信息。
2. `public.roles`：保存角色编码、角色名称和角色说明。
3. `public.user_roles`：保存用户与角色之间的多对多关系。
4. `public.departments`：保存部门层级结构。
5. `public.positions`：保存岗位信息。

其中，`users` 提供身份主体，`roles` 提供权限语义，`user_roles` 负责两者之间的绑定。部门和岗位则主要用于组织归属和后续数据范围控制。

### 6.2.6 权限控制的设计落点

若按照传统写法，这一部分通常会展开为认证类、授权类、角色服务类等多个后端对象。但在本系统中，权限控制的实际落点主要有三个：第一，前端页面根据当前用户状态控制菜单和按钮显隐；第二，PostgREST 在请求进入数据库前传入 JWT 声明；第三，数据库函数和 RLS 策略最终决定当前用户是否具备读取、写入或推进流程的资格。因此，本部分的详细设计重点不在“类的方法调用关系”，而在“请求进入后由谁校验、在哪一层拦截以及最终如何落表”。

### 6.2.7 本节小结

通过上述设计，系统已经完成了从“用户身份”到“页面访问”再到“数据库写入边界”的完整控制链路。这一部分实现了开题报告中关于 HMS 安全体系和数据库层权限控制的主要目标。

## 6.3 库存入库与出库详细设计

### 6.3.1 设计目标

库存入库与出库功能是物料管理模块中最核心的业务功能之一，直接关系到库存台账准确性、批次追溯能力和现场操作规范性。本系统在详细设计阶段重点解决了以下问题：

1. 如何在入库时自动生成或校验批次号；
2. 如何在出库时防止超卖或错误批次出库；
3. 如何保证库存批次表与库存流水表同步更新；
4. 如何让流程实例信息能够与库存单据草稿形成绑定。

### 6.3.2 页面结构设计

库存入库页面和库存出库页面均采用“表格 + 抽屉表单”的交互方式。表格区域用于展示当前库存流水或草稿数据，抽屉表单用于登记新的出入库业务。这样设计的好处是：用户在录入时能够同时参考已有数据，不需要频繁切换页面。

以入库页面为例，主要输入项包括物料、仓库、入库类型、批次号规则、批次号、数量、单位、生产日期和备注等字段；当用户启用批次号规则时，系统可调用数据库函数生成批次号；当用户关闭规则时，也可改为手工录入。出库页面则重点增加了库存分布提示和批次选择逻辑，使用户能够在出库前清楚了解某个物料在不同仓位和批次下的可用库存。

### 6.3.3 入库功能处理逻辑

入库功能的核心逻辑由前端表单收集参数后，调用数据库函数 `scm.stock_in` 完成。其处理流程如下：

1. 前端选择物料、仓库、批次号和数量等信息；
2. 若启用批次号规则，则调用 `scm.generate_batch_no` 生成批次号；
3. 前端提交入库请求；
4. 数据库函数在事务内查找是否已存在对应批次；
5. 若不存在，则创建新的库存批次记录；
6. 若已存在，则累加该批次的可用库存；
7. 同步写入库存流水表；
8. 返回入库结果、批次编号和更新后的库存数量。

该设计的关键点在于：库存批次和库存流水并不是分散处理，而是由数据库函数在同一事务中完成，从而保证库存状态和流水记录的一致性。

### 6.3.4 批次号生成函数设计

批次号生成函数 `scm.generate_batch_no` 的设计目标，是兼顾规则生成和人工输入两种场景。该函数的主要输入输出如下。

表6-2 批次号生成函数输入输出说明

| 项目 | 内容 |
| --- | --- |
| 函数名称 | `scm.generate_batch_no` |
| 输入参数1 | `p_rule_id`：批次号规则编号 |
| 输入参数2 | `p_material_id`：物料编号 |
| 输入参数3 | `p_manual_override`：手工指定批次号 |
| 返回值 | `TEXT` 类型批次号 |
| 主要作用 | 根据规则模板或人工输入生成唯一批次号 |

从实现逻辑上看，该函数先判断是否存在人工输入；若存在，则直接进行唯一性校验；若不存在，则加载规则表和物料表信息，根据模板替换物料编码、日期、分类和序号占位符，最后生成批次号并再次校验唯一性。该设计较好地适配了食品行业中“同一物料不同批次需精细区分”的需求。

### 6.3.5 出库功能处理逻辑

出库功能的核心由数据库函数 `scm.stock_out` 实现，其逻辑重点是防止超卖。总体流程如下：

1. 前端根据物料和仓库筛选可选批次；
2. 用户选定批次并填写出库数量；
3. 系统读取该批次当前可用库存；
4. 调用 `scm.stock_out` 执行出库；
5. 数据库在事务内锁定批次记录；
6. 若可用数量不足，则抛出异常并终止；
7. 若库存充足，则扣减库存批次表中的数量；
8. 同步写入库存流水表并返回出库结果。

表6-3 出库函数输入输出说明

| 项目 | 内容 |
| --- | --- |
| 函数名称 | `scm.stock_out` |
| 输入参数1 | `p_material_id`：物料编号 |
| 输入参数2 | `p_warehouse_id`：仓库编号 |
| 输入参数3 | `p_quantity`：出库数量 |
| 输入参数4 | `p_unit`：计量单位 |
| 输入参数5 | `p_batch_no`：批次号 |
| 输入参数6 | `p_transaction_no`：单据号，可为空 |
| 输入参数7 | `p_operator`：操作人 |
| 输入参数8 | `p_remark`：备注 |
| 返回值 | `JSONB`，包含执行结果、单据号、批次编号和剩余数量 |

这种设计的优点在于，库存扣减和流水登记统一在数据库层完成，避免了应用层先查库存、后写流水时可能出现的并发不一致问题。

### 6.3.6 草稿与流程绑定设计

为了支持库存业务与流程审批联动，系统在库存模块中引入了 `inventory_drafts` 草稿机制，并在前端通过 `workflowContext` 把流程实例编号、业务键、任务节点等信息写入单据草稿。这样做的意义在于：库存业务不再只是独立的出入库记录，而可以作为流程驱动场景中的业务载体。

例如，当某个库存出库单由流程入口发起时，页面会把当前流程实例编号和业务键保存到草稿属性中。后续审批中心和状态映射模块即可据此继续推进业务状态。这一设计实际上把物料模块和流程能力连接了起来，是系统详细设计中比较关键的一条链路。

### 6.3.7 库存出入库时序设计

在数据库中心架构下，库存模块最能体现“没有传统后端类但仍然可以做详细设计”的特点。前端页面只负责采集参数、显示结果和持有草稿上下文，真正的库存事务处理由数据库函数完成。因此，本文采用时序图来展示页面、接口、函数和核心业务表之间的调用顺序。

### 6.3.8 本节小结

库存入库与出库功能通过“前端表单 + 数据库函数 + 流程草稿绑定”的方式完成了详细设计。其核心价值在于：既满足了现场操作便捷性，也保证了库存批次、流水记录和流程上下文之间的一致性。

## 6.4 流程审批与状态映射详细设计

### 6.4.1 设计目标

流程审批与状态映射部分主要用于支撑应用中心中的流程设计、审批中心和业务状态联动能力。相较于把流程系统独立为外部产品，本系统更强调流程定义、流程实例和业务状态之间的一体化设计。

### 6.4.2 流程运行核心结构

当前流程运行部分主要由以下几类对象组成：

1. `workflow.definitions`：保存流程定义信息；
2. `workflow.instances`：保存流程实例运行状态；
3. `workflow.task_assignments`：保存任务节点候选角色与候选人；
4. `workflow.instance_events`：保存流程事件；
5. `workflow.task_approvals`：保存审批意见与会签记录；
6. `app_center.workflow_state_mappings`：保存流程节点到业务状态字段的映射规则。

这些对象共同形成了从“流程定义”到“流程执行”再到“业务写回”的完整链路。

### 6.4.3 审批中心页面设计

审批中心页面主要用于跨流程查看审批进度和审批意见。页面包含三个核心部分：第一部分是统计汇总，用于显示当前会签节点数、待办数量、审批记录总数等；第二部分是筛选区，用于按流程定义、关键字和待办状态进行过滤；第三部分是会签进度表和审批意见明细表，用于展示实例级进度和审批记录。

这种设计使审批中心不再只是单一流程的处理页面，而成为了一个面向多个流程实例的汇总查看页面。对于管理层或审批角色而言，这一界面能够显著提高任务查看效率。

### 6.4.4 流程实例启动与流转函数设计

流程实例的关键处理函数主要包括 `workflow.start_workflow_instance` 和 `workflow.transition_workflow_instance`。前者负责初始化流程实例，后者负责根据当前节点、审批信息和任务分派规则推进流程。

表6-4 流程实例启动函数说明

| 项目 | 内容 |
| --- | --- |
| 函数名称 | `workflow.start_workflow_instance` |
| 输入参数1 | `p_definition_id`：流程定义编号 |
| 输入参数2 | `p_business_key`：业务主键 |
| 输入参数3 | `p_initial_task_id`：初始节点编号 |
| 输入参数4 | `p_variables`：运行时变量 |
| 返回值 | `workflow.instances` 记录 |
| 主要作用 | 创建流程实例并确定起始任务节点 |

表6-5 流程节点执行权限判断函数说明

| 项目 | 内容 |
| --- | --- |
| 函数名称 | `workflow.can_execute_task` |
| 输入参数1 | `p_definition_id`：流程定义编号 |
| 输入参数2 | `p_task_id`：任务节点编号 |
| 返回值 | `BOOLEAN` |
| 主要作用 | 根据当前 JWT 角色与用户名判断是否允许执行该任务 |

从实现角度看，流程执行不是单纯依赖前端决定“谁可以审批”，而是先由数据库函数读取当前声明信息，再与 `task_assignments` 中配置的候选角色和候选人进行比较，最后决定是否允许推进流程。这使得审批约束更加严格，也更符合企业系统对权限一致性的要求。

### 6.4.5 状态映射设计

状态映射表 `app_center.workflow_state_mappings` 的作用，是把流程任务节点与业务数据状态字段绑定起来。其基本思路是：当流程实例推进到某个 BPMN 节点时，系统可根据映射配置找到目标业务表、目标状态字段和值，并将状态写回业务数据。

这种设计解决了一个很关键的问题，即流程系统和业务系统不再彼此孤立。流程审批结果能够在业务表中留下明确状态，而业务页面也能够直接读取这些状态用于展示和过滤。对于毕业论文而言，这一部分正好体现了“流程与数据协同设计”这一研究特色。

### 6.4.6 流程运行时序设计

流程模块虽然也没有传统的流程服务类和审批服务类，但其运行链路依然可以通过时序图清楚表达。本文将其拆解为“审批页面或流程配置页面、PostgREST 接口、流程函数、流程实例表、审批记录表和状态映射表”六类对象。

流程启动与状态写回时序如图6-3所示。用户从业务页面或应用中心发起流程后，前端把流程定义编号、业务键和运行变量提交给 PostgREST，随后由 `workflow.start_workflow_instance` 创建流程实例并写入初始状态。当审批节点推进时，`workflow.transition_workflow_instance` 会结合 `workflow.can_execute_task` 和任务分派规则判断当前操作者是否有权限继续执行；若允许推进，则更新 `workflow.instances`、记录审批日志，并根据 `app_center.workflow_state_mappings` 将状态写回业务表。

通过审批中心、任务分派、流程启动与状态映射等设计，系统已经完成了流程能力的核心实现，并形成了从流程定义到业务状态写回的闭环。这也实现了开题报告中关于 BPMN 流程设计与流程运行联动的主要目标。

## 6.5 轻量化语义模块详细设计

### 6.5.1 设计目标

轻量化语义模块的设计目标，是在不引入复杂推理引擎的前提下，为系统中的数据表、字段和关系提供统一的语义描述能力。其核心作用不是替代业务表，而是在现有业务数据之上构建一层“可理解的结构说明”。

### 6.5.2 模块定位

在整体系统中，轻量化语义模块位于数据支撑层与应用中心之间。一方面，它依赖数据库中的表结构、关系和元数据；另一方面，它又通过本体关系工作台和列级语义展示能力向上层提供可视化和可解释支撑。因此，这一模块既具有底层治理属性，也具有面向应用配置的支撑属性。

### 6.5.3 本体关系工作台设计

本体关系工作台是轻量化语义模块最直观的前端载体。页面主要由三部分组成：第一部分是关系指标统计，展示当前关系数量、表数量、本体关系数和外键关系数；第二部分是图关系总览区，用于可视化展示表与表之间的语义关系；第三部分是列级语义面板和关系明细列表，用于展示字段语义、敏感标记、语义来源等信息。

该页面的关键交互是“图关系聚焦”。用户可以点击某张表节点，将视图聚焦到该表相关的关系，再查看该表的列级语义详情。这种设计使得语义信息不再只是隐藏在数据库脚本中的配置数据，而是能够被系统设计者和管理人员直接观察和理解。

### 6.5.4 语义数据组织设计

当前系统中的语义能力主要体现为：

1. 表级语义：描述一张业务表在系统中的业务含义；
2. 列级语义：描述某个字段对应的业务语义、数据类型、界面类型和敏感标记；
3. 关系语义：描述表与表之间是外键关系、本体关系还是权限关系；
4. 中文化回填：为系统中已有英文表名和字段名补充中文语义标签。

这几类能力共同构成了轻量化语义模块的基础。其中，列级语义尤其重要，因为它直接关系到后续动态表单配置、字段解释和权限显示。

### 6.5.5 语义模块与业务系统的协同

轻量化语义模块并不是孤立存在的。当前系统已经在三个方向上实现了协同：

1. 与应用中心协同：数据应用配置和语义信息可以共同作用于界面字段展示；
2. 与流程配置协同：流程节点状态和业务状态可以在语义层形成更统一的解释；
3. 与权限控制协同：敏感字段和重要关系能够与权限策略形成兼容约束。

这种设计方式比传统“只做几个业务页面”的毕业设计更进一步，因为它开始关注系统的可解释性、可治理性和可扩展性。对于本章而言，这也意味着轻量化语义模块的详细设计不能只写“页面展示了什么”，还需要说明语义数据如何组织、如何与应用配置联动、如何进入流程与权限解释链路。

### 6.5.6 本节小结

轻量化语义模块详细设计的完成，使本文在一般信息系统开发基础上增加了一层语义治理能力。这一部分不仅是论文中的特色内容，也为后续应用配置和流程联动提供了更统一的底层表达。

## 6.6 本章小结

本章围绕用户与权限控制、库存入库与出库、流程审批与状态映射以及轻量化语义模块四个方面，说明了系统的详细设计。总体来看，EISCore 的详细设计并不是简单罗列页面功能，而是把前端页面、数据库函数、流程控制和语义治理组合成了一套可运行的实现链路。通过这些详细设计，开题阶段提出的核心目标已经进一步落实为具体的系统结构与处理流程，为下一章系统实现与测试分析奠定了基础。
