## 5.1 数据库设计目标

数据库设计是本课题的核心部分。与传统企业系统中数据库主要承担持久化存储不同，EISCore 采用数据库中心架构，因此数据库不仅要完成数据保存，还要承担权限控制、流程关联、状态映射和语义增强等职责。换句话说，数据库设计既是系统的数据模型设计，也是系统治理能力设计的一部分。

结合南派食品的业务需求和当前项目实现情况，本系统数据库设计主要围绕以下目标展开。第一，建立稳定的基础主数据模型，统一组织架构、用户、角色、物料、仓库等关键对象。第二，围绕库存批次、库存流水、入库出库和盘点等业务构建仓储链路，支撑批次追溯和库存透明化。第三，为应用中心和工作流运行建立独立的数据域，使动态应用、流程设计和审批能力能够在同一数据库中得到统一管理。第四，通过 RLS 和语义增强结构，为后续权限治理和本体语义描述提供底层支撑。

## 5.2 数据库总体结构设计

EISCore 数据库以 PostgreSQL 16 为核心，采用多 Schema 方式对不同业务域进行划分。当前系统中与论文主线密切相关的 Schema 主要包括 `public`、`hr`、`scm`、`app_center`、`workflow` 和 `app_data`。

其中，`public` Schema 主要保存组织结构、用户、角色、权限、物料主数据及部分系统公共表；`hr` Schema 主要承载员工档案、考勤、工资和人事扩展信息；`scm` Schema 主要承载仓库、库存批次、库存流水和盘点等供应链与仓储数据；`app_center` Schema 主要承载应用中心中的应用注册、发布路由、状态映射和执行日志；`workflow` Schema 主要承载流程定义、流程实例和任务分派；`app_data` Schema 则用于承载动态生成的数据应用表。

这种划分方式的优势主要有三点。第一，不同业务域在逻辑上更加清晰，便于后续维护和权限分层。第二，流程、应用中心和业务主数据之间虽然相互关联，但又能保持清楚的边界，避免所有表混杂在同一命名空间下。第三，配合 PostgREST 和 RLS 后，不同 Schema 能够更方便地形成权限隔离和接口边界。

## 5.3 核心数据实体关系设计

从论文主线出发，EISCore 的核心实体关系主要围绕组织权限主线、物料仓储主线和应用流程主线展开，其核心实体关系如图5-1所示。
见图5-1 [图5-1_EISCore核心数据实体关系图.svg](/home/lzr/eiscore/docs/diagrams/图5-1_EISCore核心数据实体关系图.svg)。

在组织权限主线中，`departments`、`positions`、`users`、`roles` 和 `user_roles` 构成了基本的用户组织关系。其中，部门表支持层级结构，岗位表和用户表分别通过外键关联部门，角色表则用于承载系统权限划分，用户与角色之间通过关联表建立多对多关系。该结构为系统后续的人事管理、菜单授权和字段可见范围控制提供了基础。

在物料仓储主线中，`raw_materials`、`warehouses`、`inventory_batches` 和 `inventory_transactions` 构成了库存业务的核心链路。物料主数据表定义了所有物料对象，仓库表通过树形结构描述仓库、库区和库位，库存批次表记录某个物料在某个仓库中的批次库存，库存流水表则记录所有库存变动行为。通过这几类表的组合，可以支撑采购入库、领料补料、生产入库、销售出库以及盘点等多种业务场景。

在应用流程主线中，`categories`、`apps`、`workflow_definitions`、`workflow_instances`、`task_assignments`、`published_routes` 和 `workflow_state_mappings` 构成了应用中心和工作流运行的核心结构。应用表用于登记动态应用和流程应用，流程定义表与应用表建立关联，流程实例表和任务分派表分别承载运行态和任务分派信息，而发布路由和状态映射表则分别用于前端路由发布和 BPMN 节点到业务状态的映射。

总体来看，这三条主线共同组成了 EISCore 的数据库核心骨架。它们之间虽然分属不同业务域，但并不是彼此孤立，而是通过权限、批次、状态和流程关联形成了统一的数据底座。

## 5.4 组织与权限相关表设计

### 5.4.1 部门表与岗位表

部门表 `public.departments` 用于描述组织结构，主要字段包括 `id`、`name`、`parent_id`、`leader_id`、`sort` 和状态相关字段。通过 `parent_id` 自关联，系统能够表达树形组织结构，为后续的人事管理和数据范围控制提供依据。岗位表 `public.positions` 则通过 `dept_id` 关联部门，描述部门内的岗位信息。

这种设计的优点在于组织结构既可以服务于人员归属管理，也可以服务于权限和数据范围控制。例如，仓库人员和质检人员可以归属于不同部门，在系统中通过部门和岗位信息进一步约束其可操作数据范围。

### 5.4.2 用户表、角色表与用户角色关联表

用户表 `public.users` 主要保存系统登录用户和其基本信息，关键字段包括 `username`、`full_name`、`dept_id` 和 `position_id`。角色表 `public.roles` 保存角色编码、角色名称以及所属部门等信息。由于一个用户可能拥有多个角色，因此系统通过 `public.user_roles` 建立用户和角色之间的多对多关系。

这一设计方式使系统既能支持“一个用户只有一个主角色”的简单场景，也能支持“同一用户在不同业务中承担不同角色”的复杂情况。例如，某些管理人员既可能具有审批权限，也可能具有查看经营数据的权限，这些能力可以通过不同角色组合来表达。

## 5.5 物料与仓储相关表设计

### 5.5.1 物料主数据表

物料主数据表 `public.raw_materials` 是仓储与生产链路的起点。该表主要字段包括 `id`、`batch_no`、`name`、`category`、`properties` 和 `dept_id` 等。虽然当前表名为 `raw_materials`，但在实际应用中并不只服务于原材料，而是承载了整个系统中的物料基础对象。通过 `properties` JSONB 字段，系统还能够存放部分灵活属性，为非标准化物料管理提供支持。

### 5.5.2 仓库表与仓库层级结构

仓库表 `scm.warehouses` 通过 `parent_id` 构成树形结构，可同时表示仓库、库区和库位三个层级。主要字段包括 `code`、`name`、`parent_id`、`level` 和 `properties` 等。该设计比较适合南派食品当前“一个仓库内部再细分多个区域和位置”的现实场景，也便于后续仓库布局可视化。

### 5.5.3 库存批次表

库存批次表 `scm.inventory_batches` 是本系统支撑食品行业批次管理的关键表。该表以 `material_id` 和 `warehouse_id` 为基础，记录具体批次号、可用数量、锁定数量、生产日期、过期日期以及批次状态等信息。由于食品企业对批次和保质期要求较高，因此该表在系统中承担了库存台账和追溯链条中的关键角色。

### 5.5.4 库存流水表

库存流水表 `scm.inventory_transactions` 用于记录所有库存变动行为，主要包括采购入库、生产领料、补料出库、生产入库、销售出库、盘点调整等。该表通过 `material_id`、`batch_id` 和 `warehouse_id` 与物料、批次和仓库建立关联，同时保留交易类型、数量、关联单据和操作时间等信息。

在业务实现上，库存流水表具有两个重要作用。第一，它为库存批次数量变化提供可审计的来源记录。第二，它为后续批次追溯和业务问题分析提供详细历史数据。因此，该表实际上是连接静态库存状态和动态业务行为的桥梁。

## 5.6 应用中心与流程相关表设计

### 5.6.1 应用分类表与应用表

应用分类表 `app_center.categories` 和应用表 `app_center.apps` 共同构成应用中心的基础结构。分类表主要用于区分数据应用、流程应用和其他应用类型。应用表则记录具体应用的基本信息，包括应用名称、分类、类型、状态、配置和 BPMN XML 等。

由于 EISCore 并不只是固定业务页面集合，还需要支持动态应用配置和流程设计，因此 `app_center.apps` 在数据库中扮演了“应用注册中心”的角色，是应用中心模块的核心表之一。

### 5.6.2 发布路由与状态映射表

发布路由表 `app_center.published_routes` 用于记录应用发布后的访问路径，便于前端基座动态挂载和访问。状态映射表 `app_center.workflow_state_mappings` 则负责把流程任务节点与具体业务表的状态字段进行关联，使流程推进时能够驱动业务数据状态更新。

这两类表的存在使应用中心和业务系统之间不是简单的页面跳转关系，而是形成了更紧密的运行时协同关系。

### 5.6.3 流程定义表、实例表与任务分派表

流程定义表 `workflow.definitions` 用于保存 BPMN 流程定义，并通过 `app_id` 关联应用中心中的流程应用。流程实例表 `workflow.instances` 保存流程运行中的实例状态，包括当前任务节点、业务键和状态等。任务分派表 `workflow.task_assignments` 则用于定义某个流程节点可由哪些角色或用户处理。

通过这三类表，系统能够从“静态流程定义”延伸到“动态流程运行”和“任务权限分派”，从而实现审批流和业务数据的联动。

## 5.7 数据安全与权限控制设计

EISCore 数据库设计中的一个关键特点，是将数据权限控制下沉到数据库层实现。系统在多个关键表上启用了 Row Level Security，并根据 JWT 声明中的角色信息判断当前用户是否具备访问某类数据的权限。相较于单纯依赖前端菜单隐藏或应用层 if 判断，这种设计在一致性和安全性上更有保障。

例如，应用中心中的 `apps`、`published_routes`、`workflow_state_mappings` 和 `execution_logs` 等表均配置了针对不同操作类型的 RLS 策略；工作流相关表也通过角色声明约束写入和管理权限。这种设计使系统能够在多子应用共同访问同一数据库的情况下，保持统一的数据边界控制。

## 5.8 语义增强相关设计

除了传统数据表之外，系统还在数据库层引入了轻量语义增强结构，用于支持表级语义、列级语义和关系语义描述。这部分结构并不在核心 ER 图中完全展开，但在总体设计中占有重要位置，因为它直接关系到后续的动态应用配置、字段解释和语义治理。

从数据库实现角度看，语义增强的意义主要体现在两个方面。第一，它使数据库中的表和字段不再只是技术标识，而能够附带更明确的业务含义。第二，它为后续把流程状态、权限语义和数据结构联系起来提供了更统一的表达基础。这也是本课题区别于一般 CRUD 型毕业设计的重要地方之一。

## 5.10 本章小结

本章围绕 EISCore 的数据库设计展开，重点说明了多 Schema 结构、核心实体关系、组织权限主线、物料仓储主线以及应用流程主线的设计思路。整体来看，本系统的数据库不仅承担了数据存储职能，还承担了权限控制、流程协同和语义增强等更底层的治理能力。这种数据库中心的设计方式，与课题“轻量、可扩展、低运维成本”的整体目标保持一致，也为后续详细设计和系统实现提供了稳定的数据基础。
