## 2. 系统定位

EISCore 是面向中小制造企业的 AI+企业信息化系统核心底座。当前代码库已经不只是单页演示或论文样例，而是由基座门户、多个业务子应用、数据库脚本、PostgREST API、工作流、本体语义、移动端、实时智能体运行时和部署配置共同组成的模块化工程。

系统围绕以下业务对象展开：

1. 人员、组织、用户、角色和权限。
2. 物料、批次、仓库、库区、库位和库存流水。
3. 销售客户、商机、订单、出货和回款。
4. 采购供应商、采购需求、采购订单和到货。
5. BOM、生产建议、生产工单、领料和完工入库。
6. 检验、质量异常、整改任务、审核和检验标准。
7. 设备台账、点检、异常、维保工单和保养计划。
8. 应用配置、数据应用、流程设计、审批中心和本体关系。
9. 决策驾驶舱和移动现场作业。
10. AI Copilot、Agent Runtime、知识问答和受控工具调用。

## 3. 总体架构

### 3.1 架构分层

| 层级 | 组成 | 说明 |
|---|---|---|
| 统一入口层 | Nginx、基座前端 | 通过 `http://localhost` 作为统一入口，承载主应用、子应用、API、Swagger、Agent Runtime 和 code-server 代理。 |
| 前端基座层 | `eiscore-base` | 登录、主布局、侧边栏、页签、主题、系统设置、业务流程图、AI Copilot、qiankun 容器。 |
| 微前端业务层 | HR、Materials、Apps、Mobile、Sales、Purchase、Production、Quality、Equipment、Decision | 按业务域拆分为子应用，由 qiankun 在基座中装载。 |
| 数据 API 层 | PostgREST | 将 PostgreSQL schema、表、视图和 RPC 暴露为 REST API。 |
| 数据核心层 | PostgreSQL 16 | 承载业务表、视图、函数、触发器、RLS、权限、工作流状态映射和动态应用表。 |

| 实时与智能体层 | `realtime` | WebSocket、PostgreSQL LISTEN/NOTIFY、AI Chat、Flash 工具调用、Agent Runtime、工作流引擎。 |

## 5. 登录、账号与权限

### 5.1 登录流程

后台登录由 `eiscore-base/src/views/LoginView.vue` 提供。登录请求调用：

```text
POST /api/rpc/login
```

登录成功后：

1. 服务端返回 JWT token。
2. 前端解析 token 并保存 `auth_token`。
3. 用户信息、权限列表、角色信息写入本地状态。
4. 路由守卫按 token 是否存在和是否过期决定是否进入后台。

### 5.2 移动端登录

移动端使用独立登录页面 `/mobile/login`。移动端路由也校验 `auth_token`，token 缺失或过期时跳转登录页。

### 5.3 移动端自动跳转

基座路由会检测手机/平板访问。如果不是公开产品页，移动设备会自动跳转到：

```text
/mobile/
```

公开产品介绍页 `/eiscore` 不触发移动端后台跳转。

### 5.4 权限模型

系统采用组合权限模型：

1. 用户绑定角色。
2. 角色授予权限点。
3. 权限点包括模块权限、应用权限、操作权限和状态迁移权限。
4. PostgreSQL RLS 继续作为后端数据访问控制基础。
5. 前端根据权限控制菜单、应用卡片、按钮、字段可见性和字段可编辑性。

权限码约定：

| 类型 | 格式 | 示例 |
|---|---|---|
| 模块权限 | `module:{moduleKey}` | `module:production` |
| 应用权限 | `app:{appKey}` | `app:mms_inventory` |
| 操作权限 | `op:{appKey}.{action}` | `op:hr_employee.edit` |
| 状态迁移权限 | `op:{appKey}.status_transition.{from}_{to}` | `op:mms_inventory.status_transition.created_active` |

### 5.5 字段权限

通用表格组件会读取 `public.sys_field_acl`：

1. `can_view=false` 时字段不可见或不可操作。
2. `can_edit=false` 时字段只读。
3. 字段权限按 `module` 和 `role_id` 生效。
4. 字段权限和工作流绑定字段规则叠加执行。

### 5.6 状态控制

系统使用轻量业务状态词表：

| 规范状态 | 含义 |
|---|---|
| `created` | 草稿/待处理。 |
| `active` | 生效/进行中。 |
| `locked` | 锁定/封存。 |

兼容映射：

| 历史值 | 规范值 |
|---|---|
| `draft` | `created` |
| `disabled` | `locked` |

3. 人事管理。

```text
/settings
```

1. 修改系统标题。
2. 修改主题颜色。
3. 开启或关闭通知。
4. 设置物料分类层级。
5. 配置登录门户品牌信息。

基座提供 AI Copilot 组件，作为企业经营助手和工作助手的统一交互入口。

主要能力：

1. 流式对话。
2. 多会话历史。
3. 文件上传和解析。
4. 图片、Word、Excel 等输入辅助。
5. Markdown 渲染。
6. 识别表单模板并保存到模板库。
7. 识别公式并应用到当前表格。
8. 识别表格数据并导入当前表格。
9. 识别 BPMN/流程定义并保存为流程应用。
10. 识别物料分类并导入物料分类。
11. 报表 PDF 导出辅助。

AI 历史保存在浏览器本地存储中，当前 key 为 `eis_ai_history_v5`。

### 18.4 RLS 与触发器

数据库大量使用：

1. Row Level Security。
2. 角色授权策略。
3. 更新时间触发器。

## 19. 跨模块业务链路

### 19.1 销售到生产链路

典型路径：

1. 销售管理创建销售订单。
2. 销售订单指定产品、数量、交付日期和订单状态。
3. 生产模块读取销售需求和 BOM。
4. 生产建议按销售需求减成品库存计算还需生产数量。
5. 生产建议转为生产工单。
6. 工单展开工单用料。
7. 仓储模块按生产领料单执行生产领料出库。
8. 生产模块进行订单/工单报工。
9. 仓储模块按生产入库单执行成品入库。
10. 销售出货申请进入销售出库单。
11. 仓储模块执行销售出库。
12. 销售模块登记回款。

当前状态说明：

1. 页面入口和数据对象已形成。
2. BOM、生产建议、生产工单和领料视图已接入。
3. 部分自动下推能力依赖 SQL 函数和实际数据配置。
4. 业务流程图当前用于简单链路展示和导航，不等于所有节点都已完全自动流转。

### 19.2 销售到采购链路

典型路径：

1. 销售订单产生物料需求。
2. BOM 展开识别需采购物料。
3. 生成采购需求。
4. 采购需求转采购订单。
5. 采购订单登记到货。
6. 到货记录进入 IQC 或质量检验。
7. 合格后进入采购入库。
8. 库存台账记录入库流水。

相关数据库函数：

```text
scm.create_purchase_demands_from_sales_bom(...)
```

### 19.3 采购到库存链路

典型路径：

1. 建立供应商档案。
2. 创建采购需求。
3. 创建采购订单。
4. 登记到货。
5. 记录 IQC 状态。
6. 生成或填写入库单号。
7. 仓储模块执行采购入库。
8. 库存查询和库存台账更新。

### 19.4 生产到质量链路

典型路径：

1. 生产工单执行过程中产生过程检验需求。
2. 质量模块进入生产检验。
3. 检验台账记录抽检数、不良数和结果。
4. 不合格时登记质量异常。
5. 质量异常生成整改任务。
6. 责任人完成整改。
7. 检验或验证人员关闭整改。

当前状态说明：

1. 生产检验入口已存在。
2. 检验台账、NCR 和整改任务已存在。
3. 自动从检验不合格生成 NCR 的闭环需结合当前 `quality-ncr.js` 和实际页面操作进一步验收。

### 19.5 设备点检到维保链路

典型路径：

1. 设备台账维护设备基础信息。
2. 巡检计划或保养标准定义检查内容。
3. 点检记录记录异常项。
4. 设备异常登记故障和责任归属。
5. 设备异常转维保工单。
6. 维修人员处理并记录停机时长。
7. 验收后关闭工单。
8. 设备总览更新健康评分、异常和维保情况。

### 19.6 工作流到业务状态链路

典型路径：

1. 应用中心创建流程应用。
2. 工作流设计器保存 BPMN XML。
3. 配置用户任务分派。
4. 配置节点到业务表状态的映射。
5. 用户在审批中心启动或推进流程。
6. `workflow.transition_workflow_instance(...)` 校验任务权限。
7. 流程推进后执行状态写回。
8. 写回结果记录到 `workflow.instance_events`。

适用场景：

1. 入职审批。
2. 出入库协同流程。
3. 采购审批。
4. 领料审批。
5. 自定义数据应用审批。

## 21. 部署与运行

### 21.1 Docker 统一启动

推荐在项目根目录执行：

```bash
docker compose up -d
```

或只启动 Nginx：

```bash
docker compose up -d nginx
```

查看服务：

```bash
docker compose ps
```

### 21.2 前端构建

各前端模块通常支持：

```bash
npm run build
```

基座构建示例：

```bash
cd eiscore-base
npm run build
```

子应用构建示例：

```bash
cd eiscore-materials
npm run build
```

### 21.3 开发模式

开发模式通常需要：

1. 启动基座。
2. 启动需要调试的子应用。
3. 启动 Nginx 或直接使用 Vite 端口。
4. 确认 PostgREST 和数据库可访问。
5. 确认 Agent Runtime 是否需要启动。

示例：

```bash
cd eiscore-base
npm run dev
```

```bash
cd eiscore-materials
npm run dev
```

### 21.4 数据库初始化

Docker Compose 中数据库初始化挂载顺序：

1. `env/init_roles.sql`
2. `env/db_schema_and_data.sql`
3. `sql/app_center_schema.sql`

其他业务扩展脚本位于 `sql/` 下，应按补丁说明和当前数据库状态谨慎执行。

执行 SQL 补丁时建议：

1. 先备份数据库。
2. 使用 `ON_ERROR_STOP=1`。
3. 先在测试库验证。
4. 记录执行时间和脚本版本。
5. 不重复执行非幂等脚本。

### 21.5 API 调试

Swagger UI：

```text
http://localhost/doc/
```

PostgREST API：

```text
http://localhost/api/
```

注意：

1. 请求不同 schema 时需使用 `Accept-Profile` 或 `Content-Profile`。
2. 登录后请求需携带 `Authorization: Bearer <token>`。
3. RLS 和 JWT claims 会影响查询结果。

## 22. 使用建议与验收清单

### 22.1 基础验收

1. 能打开 `http://localhost`。
2. 能打开 `http://localhost/eiscore`。
3. 能打开登录页。
4. 能登录后台。
5. 能打开各模块应用卡片页。
6. 能打开至少一个通用表格应用。
7. 能新增、编辑、搜索、导出一条测试数据。
8. 能打开 Swagger。
9. 能访问 `/agent/` 健康或配置接口。

### 22.2 权限验收

1. 超级管理员可访问全部模块。
2. 普通角色只能访问授权模块。
3. 未授权应用卡片不可进入或被隐藏。
4. 字段 ACL 生效。
5. 状态迁移权限生效。
6. PostgREST RLS 不被前端绕过。

### 22.3 业务链路验收

建议按以下顺序验收：

1. 人事：用户、角色、权限、员工档案。
2. 仓储：物料、仓库、批次规则、入库、库存查询、出库。
3. 销售：客户、订单、回款。
4. 生产：BOM、生产建议、工单、领料。
5. 采购：供应商、采购需求、采购订单、到货。
6. 质量：检验、NCR、整改。
7. 设备：台账、点检、异常、维保。
8. 决策支持：打开各驾驶舱并检查 16:9 显示。
9. 移动端：登录、扫码、盘点、出入库、标签。
10. 应用中心：新建数据应用、发布、运行。
11. 工作流：设计、发布、启动、推进、状态写回。
12. AI：打开 Copilot、上传文件、调用助手、保存模板或流程草稿。

### 22.4 数据安全验收

1. `.env` 文件不提交。
2. 生产数据库密码和 JWT secret 不使用开发默认值。
3. AI API key 不进入 Git。
4. `docs/private-review/` 不提交。
5. 真实客户资料、账号密码、服务器地址不进入公开仓库。
6. 文档公开版和证据链文件单独管理。

## 23. 当前限制与注意事项

### 23.1 功能成熟度限制

1. 当前代码库已形成多业务模块结构，但不应表述为所有模块均已成熟商用。
2. 销售、采购、生产、质量、设备等模块已有表格、驾驶舱和数据脚本，但实际项目落地需要结合真实业务数据、权限和流程继续验收。
3. 业务流程图当前偏导航和链路说明，不能替代所有后台自动下推能力的验收。
4. 移动端部分卡片仍为规划/占位状态。
5. AI 工具调用涉及写操作时必须加强确认、权限、审计和回滚机制。

### 23.2 工程运行限制

1. Vite 7 对 Node.js 版本有要求，建议使用 Node 20.19+ 或更高兼容版本。
2. 多子应用开发时需同时启动对应 Vite 服务，否则 Nginx 代理子应用会不可用。
3. Nginx 当前通过 `host.docker.internal` 代理开发态子应用，在不同系统或 Docker 环境下可能需要调整。
4. PostgREST 查询结果受 schema profile、JWT claims 和 RLS 共同影响。
5. SQL 补丁较多，生产库执行前必须梳理执行顺序和幂等性。

### 23.3 文档边界

4. 实际流程清单。
