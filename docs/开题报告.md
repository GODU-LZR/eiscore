（设计）题目	面向湛江中小制造企业的企业信息化系统核心设计与实现——以广东南派食品有限公司为例

① 研究的目的和意义

1.1 研究背景
 当前，数字化转型已成为制造业提升竞争力的关键路径。然而，湛江地区大量中小制造企业（SMEs）受限于资金投入与技术储备，信息化水平整体依然滞后。以广东南派食品有限公司（以下简称“南派食品”）为代表的农海产品深加工企业，其生产模式具有“多批次、小批量、原料属性非标”的显著特点。在实际运营过程中，物料管理（MMS）常面临台账混乱、批次追溯困难的问题；同时，人员管理（HMS）因缺乏统一的身份治理体系，导致多系统间账号割裂，权限管控存在安全盲区。 纵观现有的市场解决方案，大型 ERP 系统（如 SAP）虽然功能完善但费用高昂且实施周期长，不适合中小企业的体量；而轻量级的 SaaS 工具往往界面固化，难以满足企业在业务发展中产生的个性化定制需求（如增加特定字段或调整页面布局）。更为关键的是，中小企业普遍缺乏专业的后端运维团队，难以负担传统三层架构软件中复杂的中间件维护成本，这成为了数字化落地的一大障碍。

1.2 研究目的
 本课题旨在设计并实现一个名为 EISCore (Enterprise Information System Core) 的企业信息化底座系统。该系统拟采用前沿的“以数据库为中心（Database-Centric）”无后端架构，聚焦于解决中小企业核心域的数字化难题。 首先，系统将构建MMS与HMS双核驱动的业务基础，建立标准化的物料主数据治理与统一身份鉴权体系，利用数据库的原生能力来保障业务数据的一致性与安全性。其次，课题将引入 BPMN 2.0 标准实现业务流程的可视化编排，并尝试建设基于浏览器版 VS Code 的全栈 AI 辅助开发器，探索将自然语言需求转化为底层数据库逻辑及前端 Vue 界面配置的技术路径，从而实现从需求描述到界面交互的自动化生成。最后，在技术架构上，系统将摒弃臃肿的应用服务器层，通过 PostgREST 自动映射 API 结合微前端技术，实现系统的轻量化部署与高扩展性。

1.3 研究意义
 本课题在理论与实践层面均具有一定的探索意义。在理论层面，课题探索了“低代码前端 + BaaS (Backend as a Service)”架构在中小企业资源受限场景下的融合应用，验证了在不编写传统后端代码（No-Backend）的情况下，如何通过 AI 技术辅助生成包含用户界面与数据库逻辑的全栈应用，为软件工程领域中的低成本架构设计提供了新的实践参考。在实际应用层面，通过 EISCore 的实施，能够帮助南派食品实现物料全生命周期的数字化追踪，解决台账混乱问题；同时，架构的“去后端化”设计将极大降低服务器资源消耗与 IT 运维成本。此外，AI 开发器的引入使得企业在面对市场变化（如新增产品线）时，能通过自然语言快速生成新的管理页面与数据表，有效解决了传统软件难以快速响应业务变更的痛点。

② 国内外研究（设计）发展状况、发展水平与存在问题

2.1 国外研究现状
 在国外，Gartner 提出的“组装式企业”概念正逐步落地，强调软件系统的模块化与可重组性。在技术层面上，微前端（Micro-frontends）已成为大型 Web 应用的标准解耦方案。在架构演进方面，“Thick Database（厚数据库）”理念回归，PostgREST 等工具的成熟使得开发者开始直接利用数据库处理业务逻辑，减少中间层的性能损耗。同时，Generative UI（生成式 UI）技术正在兴起，结合 Vercel v0 等工具，利用 AI 直接生成可用的前端组件代码已成为软件工程领域的新趋势。

2.2 国内研究现状
 国内制造业信息化正处于从“单体应用”向“云原生”过渡的阶段。然而，大部分面向中小企业的定制软件仍采用传统的 Java 或 Node.js 后端架构，导致代码量大、维护困难。在定制化开发方面，虽然低代码平台（Low-Code）逐渐流行，但大多数平台基于私有协议，无法生成标准的 Vue 或 React 代码，导致企业容易被厂商锁定，无法进行深度的二次开发与集成。

2.3 存在的问题
 综合分析现状，当前针对中小企业的定制化开发主要存在三个问题：第一，架构过重导致运维困难，传统架构需要维护复杂的应用服务器，这对缺乏专业IT人员的中小企业是巨大负担；第二，前端定制僵化，现有系统的界面通常是固定的，企业想要调整表单布局或增加展示列，往往需要原厂开发介入，周期长且费用高；第三，业务与代码割裂，前后端分离架构导致修改一个简单的业务功能往往需要同时调整前端代码、后端接口和数据库，协同成本较高。

③ 研究（设计）主要内容、预期目标及拟解决的关键问题

3.1 主要研究内容
 本课题将构建基于无后端架构的 EISCore 系统，主要研究内容包含以下五个方面： 第一，建设 MMS 物料管理核心模块。 利用 PostgreSQL 的 JSONB 特性灵活存储南派食品差异化的原料属性，解决非标品管理难题；同时将采购入库、领退料等核心业务逻辑封装为数据库存储过程，确保台账变更的原子性。 第二，构建 HMS 原生安全体系。 建立支持 OpenID Connect (OIDC) 协议的统一认证中心，并摒弃应用层代码鉴权，直接利用数据库的 RLS (Row Level Security) 策略实现细粒度的授权，从底层保障数据安全。 第三，部署自动 API 网关。 引入 PostgREST 中间件，自动将数据库的表结构和逻辑函数映射为标准的 RESTful API，供前端直接调用，从而免去传统后端接口的开发工作。 第四，集成 BPMN 2.0 流程引擎。 整合前端流程设计器，使流程的状态流转能够直接触发底层的数据库逻辑函数，实现业务流程的闭环管理。 第五，探索全栈 AI 开发器。 在服务端容器中运行 WebIDE 并集成大模型推理能力。重点研究如何将自然语言需求（如“增加水果糖度检测环节”）自动转化为 SQL DDL 脚本与 PL/pgSQL 业务函数，并尝试基于 Element Plus 组件库自动生成对应的 Vue.js 页面代码与 JSON 表单配置，实现基于自然语言的需求即时响应。

3.2 预期目标
 课题预期完成一套无后端应用服务器的 EISCore 系统设计与编码，并在南派食品进行原型验证。在性能方面，得益于链路缩短，关键业务接口 API 响应时间预期控制在 P95 < 200ms 以内。在功能方面，系统需实现物料台账的精准追溯，并完成基于 AI 自动生成可运行的增删改查（CRUD）页面与数据库逻辑的原型验证。最终工程产出包括系统源代码、数据库脚本、用户操作手册及不少于 1.5 万字的毕业论文。

3.3 拟解决的关键问题
 本课题拟解决的关键技术问题主要包括：首先是逻辑封装难题，在摒弃传统 Service 层后，需要解决如何优雅地在数据库函数中处理复杂的跨表事务与异常回滚；其次是AI 生成的端到端一致性，需要确保 AI 生成的前端界面代码（Vue 组件）与后端数据库架构（Schema）保持严格的映射关系，避免因字段类型不匹配导致系统崩溃；最后是微前端与 API 的协同，在 qiankun 微前端架构下，需要设计通用的前端请求适配器，以实现与 PostgREST 自动生成接口的无缝对接。

④ 研究（设计）方案与技术路线

4.1 技术架构
 本系统采用 Serverless 风格的纯前端架构。前端层基于 Vue.js 3、TypeScript 和 Element Plus 技术栈，使用 qiankun 构建微前端基座，并引入动态渲染引擎以支持解析 AI 生成的配置代码。API 网关层引入 PostgREST，负责处理 HTTP 请求并将其转译为 SQL 语句。核心逻辑层以 PostgreSQL 数据库为核心，承担数据存储、PL/pgSQL 业务逻辑处理以及 RLS 安全控制。开发工具层则基于 WebIDE 集成 AI Copilot，提供从自然语言到全栈代码的自动化生成环境。

4.2 技术路线图
 项目的实施将遵循以下路线：首先进行需求分析，调研南派食品业务并输出 BPMN 流程图与数据模型；其次进行数据库核心开发，设计 Schema、编写存储过程并配置 RLS 安全策略；紧接着进行API 映射与测试，配置 PostgREST 并自动生成接口文档；随后搭建前端框架，构建 qiankun 基座及通用的动态表单组件库；在此基础上攻关AI 开发器，调试提示词工程以实现 SQL 与 Vue 代码的同步生成；最后进行系统集成测试，验证从页面交互到数据库逻辑触发的全链路连通性。

⑤ 研究（设计）方法

本课题主要采用四种研究方法：文献研究法，深入研究 Database-Centric 架构、Generative UI 技术及 BPMN 2.0 规范，为设计提供理论支撑；案例分析法，以南派食品的实际痛点为依据，设计贴合中小企业需求的轻量化方案；原型迭代法，采用敏捷开发模式，利用 AI 开发器快速生成原型，在现场与用户确认后即时修改，确保系统可用性；实证研究法，通过对比传统手写代码与 AI 生成代码在交付速度与质量上的差异，验证本课题所提架构的高效性。
