## 2. 核心判断

这次检测结果说明，当前稿件并不是“整篇都像 AI”，而是若干高敏感区存在明显的结构化说明文模板特征。

最主要的问题不是术语多，而是：

1. 摘要、绪论、章节安排的写法过于标准化。
2. 数据库设计中字段说明和表结构说明重复度高。
3. 详细设计、实现、测试章节中固定开头太多。
4. 测试表和文字说明重复复述。
5. 致谢使用了常见感谢模板。

## 3. 两份报告的结果差异

同一稿件在两套工具中的结果分别为：

1. `12.7%`
2. `34.1%`

这说明：

1. 检测器之间差异很大。
2. 单个分数不能直接作为定论。
3. 更合理的做法是按高风险段落类型做合规修订。

## 4. 高风险段落类型

### 4.1 摘要、绪论、章节安排

原因：

1. 通用背景句多。
2. 结构太平滑。
3. 章节介绍过长，且模式固定。
4. 英文摘要套话密度高。

### 4.2 数据库设计

原因：

1. 反复使用“该表用于……”句型。
2. 字段说明高度规则化。
3. 正文重复解释表格内容。
4. 设计取舍说明不够集中。

### 4.3 详细设计、实现、测试

原因：

1. 固定开头反复出现。
2. 同一段同时堆背景、流程、作用、结论。
3. 枚举和参数说明过密。
4. 测试表格和结论重复写两遍。

### 4.4 致谢

原因：

1. 常见感谢模板过强。
2. 泛化感谢词太多。
3. 项目细节不足。

## 5. 合规降低误判的处理方法

1. 先改摘要和绪论。
2. 数据库章节少念表，多写设计取舍。
3. 测试章节少写流水账，多写链路结论。
4. 固定开头要打散。
5. 增加项目锚点，例如 `qiankun`、`PostgREST`、`RLS`、`inventory_batches`、`workflow.instances`。
6. 英文摘要单独人工改。
7. 致谢改成事实型感谢。

## 6. 不建议的做法

1. 故意口语化。
2. 故意写错别字或病句。
3. 大规模同义词替换。
4. 删除必要术语、引用和真实边界。
5. 夸大未完成模块的完成度。

## 7. 最终原则

真正有效的方向不是对抗检测，而是：

`降模板感，增项目感；降复述感，增设计取舍；降结构重复，增事实锚点。`

## 8. 新平台补充分析

你新提供的报告文件为：

1. `87d1bd17-a9bd-4c9a-8d00-8f71468b8a12.docx`

报告首页给出的整体结果为：

1. `AI生成概率：63.24%`
2. `人工撰写概率：35.68%`
3. `疑似AI生成概率：1.08%`

这套平台与前面两套工具不同，它不是只给总分，而是按段落逐条标注“AI生成”或“人类撰写”。因此，它更容易把局部高模板化段落放大出来。

### 8.1 平台为什么会给高分

从逐段结果看，分数偏高的核心原因主要有 4 类。

#### 8.1.1 宏观背景段过于标准

例如设计总说明开头、绪论背景开头、研究价值开头，都采用了典型的“行业背景 - 企业困难 - 方案提出 - 研究意义”骨架。
这种写法逻辑没错，但对检测器来说可预测性很高，因此容易被判高。

#### 8.1.2 研究现状与章节安排过于论文模板化

国内外研究现状、本文主要研究内容、本文组织结构、本文特点与创新点，这几类内容本身就容易写成标准说明文。
一旦使用了“已有研究普遍认为”“总体而言”“第一章为……第二章介绍……”这类固定骨架，平台就更容易判为 AI 生成。

#### 8.1.3 第二章技术介绍太像教材式综述

这次报告里，Vue 3、Vite、微前端、组件库、PostgreSQL、PostgREST、BPMN、容器化环境等段落多数都被判高。
原因不是技术术语多，而是这些段落更像“技术是什么、有什么优点”的标准说明，而不是“它在本项目中为什么这样用、承担了什么职责”的工程化表述。

#### 8.1.4 高风险段落普遍缺少局部项目锚点

同一份报告里，被判低分的段落通常都有明显工程锚点，例如：

1. `Docker Compose`
2. `PM2`
3. `qiankun`
4. `Schema`
5. `JSONB`
6. `RLS`
7. 当前已完成的模块范围

这反过来说明，被判高分的段落主要问题不是“语言不自然”，而是“项目事实密度不够，像标准论文章节说明”。

### 8.2 新平台的判定特征

这套平台至少体现出 3 个特征：

1. `按段落局部风格判断`
   - 相邻两段只要一段更空泛、一段更具体，分数就可能从 `99%+` 跳到 `1%` 左右。
2. `对标准学术综述更敏感`
   - 绪论、研究现状、章节安排、技术介绍是这次最典型的高风险区。
3. `对工程事实更宽容`
   - 只要段落中出现明确的模块边界、表结构职责、部署链路和设计取舍，分数通常明显下降。

### 8.3 对当前修订版 5 的意义

当前文件 `docs/毕业论文总稿_按校模板终稿_修订版5_2026-03-15.docx` 的整体结构已经比较稳定，目录和章节框架完整。
但从这套平台的判定特点看，后续如果还要继续降误判风险，重点不应再放在整篇通改，而应放在以下段落类型的局部收口：

1. `设计总说明`第一段
2. `第一章`中的背景、研究现状、主要研究内容、组织结构、创新点
3. `第二章`中的技术定义段和环境说明段
4. 任何“总括性章节导读”或“标准章节小结”

### 8.4 对应的合规处理原则

针对新平台，更合适的处理方法是：

1. 少改“词”，多改“段落信息结构”。
2. 少写通用意义判断，多写项目里的具体问题、边界和设计取舍。
3. 技术介绍不要停留在“技术是什么”，而要尽快落到“为什么选它、在本项目中干什么”。
4. 章节安排和章节小结只保留必要信息，避免写成长段模板化导读。
5. 优先逐段处理高风险区，不需要整章重写。
